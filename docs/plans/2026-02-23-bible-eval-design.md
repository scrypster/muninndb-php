# Bible Eval Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a real, reproducible evaluation harness for MuninnDB using the King James Bible and public domain cross-reference data as ground truth, proving both retrieval quality and the value of cognitive memory mechanics.

**Architecture:** Two-phase eval — Phase 1 measures retrieval quality (NDCG@10 against known cross-references), Phase 2 demonstrates that Ebbinghaus decay and Hebbian co-activation measurably improve retrieval in the expected direction. A data setup script fetches the corpus on first run; Makefile targets make it one-command reproducible.

**Tech Stack:** Go (same as existing `cmd/eval/`), KJV Bible JSON (public domain), OpenBible cross-references TSV (public domain, CC0), ONNX local embedder.

---

## Background

The existing synthetic eval (`cmd/eval/`) uses template-generated memories with thin vocabulary. It measures keyword recall, not semantic relevance. It is useful as a sanity check but cannot prove whether MuninnDB's cognitive layer improves retrieval quality.

This eval replaces keyword recall with real ground truth: 340,282 verse-to-verse cross-references from OpenBible. If verse A cross-references verse B, they are semantically related by definition. We can measure whether the system surfaces B when queried with A's content.

## Corpus Structure

Each Bible verse becomes one engram:

```
Concept:  "John 3:16"
Content:  "For God so loved the world, that he gave his only begotten Son,
           that whosoever believeth in him should not perish, but have
           everlasting life."
Tags:     ["New Testament", "John", "gospel", "salvation"]
Vault:    "bible"
```

Tags are derived from:
- Testament: "Old Testament" or "New Testament"
- Book name (e.g. "John", "Genesis", "Psalms")
- Genre bucket: "gospel", "epistle", "prophecy", "wisdom", "law", "history", "poetry"

**Corpus size and load times (local ONNX, ~11 writes/sec):**

| Mode | Verses | Est. load time |
|------|--------|----------------|
| Default (NT only) | ~7,957 | ~12 min |
| Full (entire Bible) | ~31,102 | ~47 min |

Default mode loads New Testament only. `--full` flag loads entire Bible.

## Phase 1: Retrieval Quality

**Goal:** Measure whether the system surfaces cross-referenced verses when queried.

**Method:**

1. Select 100 seed verses spread evenly across NT books (1-2 per book).
2. For each seed verse, query MuninnDB using the verse text as `Context`.
3. Retrieve top-10 results.
4. Count how many of the seed verse's known cross-references appear in top-10.
5. Compute Recall@10 and NDCG@10.

**Baseline comparison:** Run each query twice:
- `--hnsw-only` mode: bypass decay scoring, Hebbian boost, confidence weighting — pure vector similarity
- Full pipeline: normal activation

This directly answers: *does the cognitive layer help or hurt retrieval quality on a cold corpus?*

**Seed verse selection:** verses with 5+ known cross-references (ensures meaningful ground truth). Pre-selected and committed to the repo as `testdata/bible/eval-seeds.json` (generated by setup script).

---

## Phase 2: Cognitive Properties

**Goal:** Show that Ebbinghaus decay and Hebbian co-activation measurably change retrieval behavior in the expected direction.

**Steps:**

1. **Baseline**: Record NDCG@10 for 10 fixed thematic queries (e.g. "forgiveness and redemption", "eternal life", "shepherd and flock", "resurrection of the dead"). Committed to repo as constants in code.

2. **Reading session**: Activate 200 verses from the Gospel of John sequentially via the full activation pipeline. This fires Hebbian co-activation links between John's verses and increments their access counts.

3. **Re-query**: Same 10 thematic queries. Measure NDCG@10 delta. Hypothesis: John-adjacent results score higher.

4. **Decay simulation**: Set `LastAccess` to 30 days ago for Old Testament genealogy verses (Numbers 1-10, 1 Chronicles 1-9 — ~450 verses of census/genealogy). Submit them to the decay worker. Re-query same 10 themes.

5. **Final measure**: Do John-adjacent results improve after reading? Do genealogy verses surface less after decay?

**Why this works as a demo:** The story is intuitive. "I read through John recently, so my memory of John's themes is fresh. I haven't touched the genealogies in a month, so they've faded." That's the thesis of the whole system, made concrete.

## Reproducibility

### Setup script

```bash
scripts/eval-bible-setup.sh
```

Downloads KJV JSON and cross-references TSV into `testdata/bible/`. Safe to re-run (idempotent). Generates `testdata/bible/eval-seeds.json` from cross-reference data.

### Makefile targets

```makefile
eval-bible:        ## Run Bible eval (New Testament, ~12 min)
eval-bible-full:   ## Run Bible eval (full Bible, ~47 min)
eval-bible-setup:  ## Download corpus data
```

### Results file

`--results-file docs/eval-results/YYYY-MM-DD-bible-eval.txt` — written automatically by `make eval-bible`. Commit to track improvement over releases.

### gitignore

```
testdata/bible/*.json
testdata/bible/*.tsv
testdata/bible/*.csv
```

(The setup script and `eval-seeds.json` are committed; raw corpus files are not.)

## Out of Scope

- Streaming/incremental loading (load all verses up front)
- Multi-translation support (KJV only)
- Chapter-level or passage-level granularity (verse only)
- Automated CI execution (manual, on-demand)
